## Karpathy-Inspired AI Coding Rules

这些规则来自 `forrestchang/andrej-karpathy-skills` 的核心思路，当前项目统一采用。  
目标是减少 AI coding 中常见的四类问题：

- 静默假设
- 过度设计
- 顺手乱改无关代码
- 没有明确验证标准就直接开写

### 1. Think Before Coding
先想清楚，再写代码。

要求：
- 不要默认替用户做未经确认的假设
- 有歧义时，先明确你采用的是哪种理解
- 有多个实现方向时，先说明取舍
- 如果发现信息不足、上下文冲突或规则不清，先指出问题
- 不要带着糊涂继续往下写

在这个项目里，尤其要先确认：
- 当前主语到底是 `Product`、`TransactionRecord` 还是 `OrderLine`
- 这次改动是“语义重构”还是“页面重写”
- 是保留兼容层，还是直接切换命名

### 2. Simplicity First
只写解决当前问题所需的最小代码。

要求：
- 不要添加用户没要求的功能
- 不要为一次性场景提前设计复杂抽象
- 不要为了“未来可能会用”而增加配置层、适配层、泛型层
- 不要引入不必要的错误处理、状态机或中间结构
- 如果 50 行能解决，就不要写成 200 行

在这个项目里，尤其要避免：
- 还没进入真实后端联调，就提前做复杂领域模型
- 还没需要客户中心页面，就提前做过度完整的 CRM 体系
- 还没改路由，就先大规模重命名全部目录结构

### 3. Surgical Changes
只改必须改的地方，不要顺手乱动。

要求：
- 不要顺手“优化”任务无关的代码、注释、样式和格式
- 不要重构没坏的东西
- 尽量匹配现有代码风格，而不是强行按自己的偏好改
- 如果发现无关的死代码或其他问题，可以指出，但不要顺手删
- 只清理“因为本次改动而新增的无用代码”

在这个项目里，尤其要注意：
- 当前任务是把系统主语从“订单为主”扶正为“商品行为主”
- 不要借这个机会把产品管理、全局布局、主题样式一起重做
- 不要在没被要求的情况下，把 `/orders` 直接改成 `/tasks`

### 4. Goal-Driven Execution
把任务转成可验证目标，再循环执行直到验证通过。

要求：
- 不要只接受“改一下”“优化一下”这种模糊指令
- 先把任务转成明确目标和验收标准
- 多步骤任务先给一个简短计划
- 每一步都要有验证方式
- 最后必须说明是否达成目标，而不是只说“已修改”

推荐格式：
1. [步骤] → verify: [验证方式]
2. [步骤] → verify: [验证方式]
3. [步骤] → verify: [验证方式]

在这个项目里，推荐这样执行：
1. 统一类型命名 → verify: 类型文件能表达 `Customer / TransactionRecord / OrderLine`
2. 重构 mock → verify: 能演示一笔交易两条商品任务
3. 重写文档 → verify: README / routes / UI / mock 文档口径一致
4. 页面语义调整 → verify: `/orders` 被理解为商品任务中心

## 强制业务口径

后续所有 AI coding、文档改写、页面实现，都必须遵守以下定义。

### 1. Product
产品模板。  
用于维护标准资料、规格明细、价格规则、定制规则、生产参考和文件资料。

### 2. Customer
客户主档。  
用于沉淀客户长期信息、历史交易、历史商品任务和售后记录。

### 3. TransactionRecord
一次购买行为的公共信息。  
用于保存平台订单号、客户收件信息、支付时间、整单备注等共享信息。

### 4. OrderLine
单件商品任务。  
这是系统真正的主操作对象。

一件商品对应一条独立商品任务。  
同一次交易中的多件商品，必须拆成多条独立商品任务分别推进。

### 5. ProductSnapshot
商品任务引用产品时保留的来源快照。  
用于核对商品任务参数与来源产品模板参数。

## 强制数据建模规则

### 1. 产品模板和商品任务不能混
禁止把 `Product` 直接当成业务执行对象使用。

### 2. 交易记录和商品任务不能混
禁止把单件商品的执行字段塞回 `TransactionRecord`。

### 3. 物流默认关联商品任务
物流记录必须优先关联 `orderLineId`。  
交易记录页只做汇总展示和统一入口。

### 4. 售后默认关联商品任务
售后记录必须优先关联 `orderLineId`。  
支持同一交易中的单件商品独立进入售后。

### 5. 状态驱动以商品任务为准
页面待办、列表筛选、进度推进，应优先基于 `OrderLine.status`。  
`TransactionRecord.aggregateStatus` 只做聚合展示，不作为主流程驱动状态。

### 6. 同一交易中的商品任务必须允许分开推进
必须支持：
- 不同状态
- 不同交期
- 不同物流
- 不同售后
- 不同负责人

## AI Coding 工作规则

### 1. 先文档，后编码
开始写页面、组件、mock、类型之前，优先确认：

- 当前阶段目标
- 页面范围
- 数据对象
- 字段语义
- 主链路
- 验收标准

如果这些没有确认，优先补文档，不要直接开写复杂代码。

### 2. 先类型，后逻辑
所有核心对象必须先定义类型，再写页面逻辑和交互逻辑。

当前阶段的核心对象至少包括：

- `Product`
- `Customer`
- `TransactionRecord`
- `OrderLine`
- `ProductSnapshot`
- `QuoteResult`
- `LogisticsRecord`
- `AfterSalesCase`

禁止先写一堆页面，再临时发明字段名和对象结构。

### 3. 单文件单职责
每个文件尽量只承担一种主要职责。

例如：
- 类型文件只定义类型
- mock 文件只放 mock 数据
- 页面文件只组织页面结构
- 业务计算函数单独抽离
- 组件文件不要同时承担页面容器 + 复杂业务计算 + mock 拼装

### 4. 一次只推进一个明确步骤
推荐顺序：

1. 类型
2. mock
3. 页面骨架
4. 组件接线
5. 局部交互
6. 自动报价
7. 文档回写
8. 清理与 review

除非明确要求，不要在一次改动里同时大改：
- 类型
- 页面结构
- 路由
- 组件命名
- 业务逻辑
- 文档口径

### 5. 主链路优先，边缘能力后置
当前阶段优先保证主链路跑通：

**产品维护  
→ 商品任务引用产品  
→ 选择规格  
→ 自动带出规格参数  
→ 自动带出基础价格  
→ 叠加固定加价规则  
→ 生成系统参考报价**

以下能力可以先占位，不急着做复杂实现：
- 客户中心完整页面
- 复杂物流系统
- 售后完整流转
- 文件高级管理
- 审批流
- BI 报表
- 真实后端联调

### 6. 先 mock 跑通，再谈真实接口
当前阶段以前端 UI 和 mock 联调为主。

每次新增页面或能力时，优先保证：
- 类型已定义
- mock 数据可用
- 页面能展示
- 主交互能演示

不要擅自扩展复杂后端接口、数据库设计或服务拆分。

### 7. 不得擅自改业务主语
当前项目的系统主语已经固定：

- `OrderLine` = 主操作对象
- `TransactionRecord` = 归组对象
- `Customer` = 沉淀对象
- `Product` = 模板对象

禁止再把整单写回唯一主操作对象。  
禁止把 `Product` 直接当作实际执行对象。  
禁止把单件商品的执行字段重新塞回 `TransactionRecord`。

### 8. 改字段必须同步改文档
如果修改以下任一内容：

- 核心对象命名
- 核心字段语义
- 页面主语
- mock 结构
- 路由语义

必须同步检查并更新：

- `README.md`
- `routes-and-pages.md`
- `ui-structure.md`
- `mock-data-schema.md`
- `handoff.md`

### 9. 改类型必须同步检查影响面
如果修改类型文件，必须同步检查：

- mock 数据
- 页面 props
- 列表字段
- 详情区块
- 组件参数
- 自动报价逻辑
- 文档示例

### 10. 优先保留兼容层，不要一次性改爆
当前项目允许分阶段过渡。

例如：
- `Order` 过渡到 `TransactionRecord`
- `OrderItem` 过渡到 `OrderLine`
- `SourceProductSnapshot` 过渡到 `ProductSnapshot`

第一阶段可以保留兼容导出层，避免一次性把所有页面改坏。  
但新代码不得继续扩大旧命名的使用范围。

### 11. 页面改造先改语义，再改路径
当前阶段允许：
- 先保留 `/orders`
- 先保留页面目录名 `orders`
- 先保留部分旧组件命名

但必须先把页面语义改正：
- `/orders` = 商品任务中心
- `/orders/new` = 新建交易记录页
- `/orders/:orderId` = 交易记录详情页

### 12. 组件优先复用，避免重复造轮子
能复用的结构尽量复用，例如：
- 页面标题区
- 摘要卡
- 筛选条
- 详情信息卡
- 商品任务卡区块
- 状态标签
- 报价展示卡
- warning 区块

### 13. 先做查看态，再做复杂编辑态
首轮页面优先保证：
- 看得清
- 结构对
- 主链路能演示
- mock 数据能跑

复杂编辑交互、复杂抽屉、复杂校验，可以后补。

### 14. 重要逻辑单独抽函数
以下逻辑不要散落在 JSX 里：
- 自动报价计算
- 商品任务状态映射
- 风险标签判断
- 列表行数据拼装
- 模板值与商品任务值对比

### 15. 每轮修改后要做最小自检
每次提交前，至少确认：
- 类型是否还一致
- mock 是否还能驱动页面
- 列表是否还能展示
- 商品任务卡是否还能正常显示
- 自动报价是否还能跑通
- 文档是否需要同步更新

### 16. 每阶段都要留 handoff
每完成一轮明显改动，必须补一份 handoff，至少写清楚：
1. 这轮改了什么
2. 新增了哪些文件
3. 改了哪些命名
4. 哪些地方还保留兼容层
5. 下一步该做什么
6. 当前还有哪些风险或未完成项

### 17. 优先做小而稳的提交
推荐按以下粒度拆任务：
- 一次只改类型
- 一次只改 mock
- 一次只改一个页面文案与字段
- 一次只改一个主区块
- 一次只改一份文档或一组强关联文档

### 18. 定期做 AI review / 清理
每完成几个模块，至少做一次 review，重点检查：
- 是否出现重复类型
- 是否出现旧命名和新命名混杂失控
- 是否出现无用 mock
- 是否出现无用组件
- 是否出现字段语义漂移
- 是否出现“整单”和“商品任务”混写

### 19. 当前阶段输出要求
AI 每次完成任务后，至少要说明：
1. 改了哪些文件
2. 为什么这样改
3. 兼容层保留了什么
4. 还没做的部分是什么
5. 下一步建议做什么

### 20. 核心执行原则
当前项目不是“一次性写完整 ERP”，而是：

**先统一对象  
→ 再统一类型  
→ 再统一 mock  
→ 再统一页面语义  
→ 再逐步增强交互**

AI coding 必须服从这个顺序。

## 核心一句话

这个项目当前不是“整单为主”的传统订单中心。
当前系统应以 **OrderLine（商品任务）** 为主操作对象，以 **TransactionRecord（交易记录）** 为归组对象，以 **Customer（客户）** 为沉淀对象，以 **Product（产品）** 为模板对象。
