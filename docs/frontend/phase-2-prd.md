# ERP Phase 2 PRD（业务流转版）

## 1. 文档定位

本文档用于定义 ERP 第二轮建设范围。

Phase 1 已完成的核心方向是：

- 以前端 UI 为主
- 以订单中心、产品管理为核心模块
- 先跑通「产品维护 → 订单引用产品 → 选择规格 → 自动带参数/价格 → 固定加价 → 生成系统参考报价」主链路

Phase 2 不推翻 Phase 1，而是在现有基础上继续扩展，目标是：

- 让订单具备真实业务流转能力
- 让订单中的待办事项结构化并可分派
- 让产品管理从“报价模板”升级为“标准产品档案”

本阶段仍然以前端实现、mock 数据、状态管理和页面结构完善为主，不进入完整后端、完整权限、财务中心、工厂协同中心等复杂阶段。

---

## 2. Phase 2 目标

Phase 2 只解决一个核心问题：

**让系统从“能录入产品、能创建订单、能算参考价”，升级为“订单能流转、任务能分派、产品资料能支撑业务执行”。**

拆解后共有 3 个目标：

### 2.1 目标 A：订单中心升级为业务流转中心

让订单不再只是静态信息页，而是具备：

- 状态流转
- 负责人
- 订单商品多行结构
- 报价调整信息
- 操作记录
- 时间线

### 2.2 目标 B：新增任务中心

把订单中的待处理事项结构化，形成：

- 可创建任务
- 可查看任务列表
- 可进入任务详情
- 可标记任务状态
- 可关联订单与订单商品

### 2.3 目标 C：产品管理升级为产品标准档案

让产品不再只是“报价模板”，而是承载：

- 品类
- 材质
- 工艺
- 规格
- 图片
- 文件
- 生产注意事项
- 售后说明

## 4. 核心业务流程

Phase 2 统一主流程如下：

1. 在产品中心维护标准产品资料
2. 新建订单
3. 在订单中引用产品
4. 生成订单商品快照
5. 自动带出规格、参数与系统参考报价
6. 根据业务需要发起任务
7. 各岗位处理任务
8. 订单随任务推进完成状态流转
9. 订单进入完成、取消或售后中状态

一句话概括：

**产品中心为订单提供标准资料，订单中心承接业务对象，任务中心推动业务落地。**

## 5. 模块规划

# 5.1 订单中心（Phase 2 重点增强）

## 5.1.1 模块目标

订单中心在 Phase 2 的目标是：

**从“订单录入页 + 订单展示页”升级为“订单流转页 + 订单执行页”。**

订单需要同时承载：

- 客户信息
- 产品信息
- 报价信息
- 任务信息
- 操作信息

---

## 5.1.2 订单状态体系

建议采用以下最小状态体系：

- `draft` 草稿
- `pending_confirm` 待确认
- `pending_design` 待设计/建模
- `pending_production_prep` 待生产准备
- `pending_shipping` 待发货
- `after_sales` 售后中
- `completed` 已完成
- `cancelled` 已取消

### 状态说明

#### 草稿
订单资料未填完整，仅内部可编辑。

#### 待确认
客户信息、产品信息、基础报价已形成，等待内部确认。

#### 待设计/建模
订单已经明确需要设计、改图或建模处理。

#### 待生产准备
设计或规格确认完成，准备进入生产协同前阶段。

#### 待发货
订单已具备发货条件，本阶段只做状态占位，不引入完整物流模块。

#### 售后中
订单已进入返修、补发、改款等售后阶段。

#### 已完成
订单正常完成。

#### 已取消
订单终止。

---

## 5.1.3 订单核心字段

### 基础信息字段
- 订单 ID
- 订单编号
- 订单状态
- 客户名称
- 客户联系方式
- 来源渠道
- 下单时间
- 交期
- 紧急程度
- 跟单负责人
- 备注

### 业务字段
- 是否需要设计 / 建模
- 是否有特殊工艺
- 是否进入售后

- 最近任务摘要

### 记录字段
- 创建人
- 创建时间
- 最后修改人
- 最后修改时间

---

## 5.1.4 订单商品结构

Phase 2 起，一个订单允许挂载多个订单商品。

每个订单商品应独立保存以下信息：

- 订单商品 ID
- 来源产品 ID
- 来源产品快照
- 商品名称快照
- 品类快照
- 选中规格
- 自动带出的参数
- 自动带出的基础价
- 固定加价项列表
- 系统参考报价
- 人工调整金额
- 人工调整原因
- 调整后展示报价
- 商品备注
- 关联任务数

### 设计原则
- 订单商品必须保留来源产品快照
- 即使后续产品资料被修改，历史订单商品也不应被直接覆盖
- 后续任务允许关联到某个具体订单商品，而不是只能关联整个订单

---

## 5.1.5 报价机制（轻量版）

Phase 2 不做复杂审批流，也不做完整多版本报价历史，但允许增加轻量报价调整字段：

- 系统参考报价
- 人工调整金额
- 调整后报价
- 调整原因
- 调整人
- 调整时间

### 设计原则
- 系统参考报价始终保留
- 人工调整只作为附加层，不覆盖系统原值
- 所有人工调整都应进入时间线记录

---

## 5.1.6 订单详情页新增模块

订单详情页建议新增以下区域：

### A. 状态流转区
展示当前状态、可切换状态、负责人、交期、紧急程度。

### B. 订单商品区
展示多行订单商品，每一行可查看：
- 来源产品
- 规格
- 参数
- 报价
- 调整信息
- 关联任务数

### C. 任务摘要区
展示当前订单相关任务：
- 任务标题
- 任务类型
- 状态
- 负责人
- 截止时间

### D. 时间线区
展示订单关键记录：
- 创建订单
- 引用产品
- 修改规格
- 重新计算报价
- 发起任务
- 状态变更
- 任务完成
- 备注更新

# 5.2 任务中心（Phase 2 新增模块）

## 5.2.1 模块目标

任务中心不是独立 OA 系统，而是 ERP 内部的业务执行驱动层。

它的目标是：

**把订单中的待办事项结构化，并可分派给具体角色。**

---

## 5.2.2 任务中心页面

### 页面 1：任务列表页
路径建议：`/tasks`

作用：
- 展示任务总表
- 按状态、分类、负责人、到期时间筛选
- 快速进入任务详情

### 页面 2：任务详情页
路径建议：`/tasks/:taskId`

作用：
- 查看任务详细信息
- 查看来源订单与来源商品
- 更新任务状态
- 查看任务记录
- 编辑备注

---

## 5.2.3 任务分类

Phase 2 仅做 4 类任务：

### A. 订单处理任务
示例：
- 补充客户资料
- 确认规格
- 确认报价
- 确认交期

### B. 设计建模任务
示例：
- 出设计图
- 改图确认
- 上传建模文件
- 确认特殊需求说明

### C. 生产准备任务
示例：
- 确认材质
- 确认工艺
- 整理生产备注
- 准备生产资料

### D. 售后任务
示例：
- 返修确认
- 补发处理
- 改款确认
- 售后跟进

---

## 5.2.4 任务状态

任务状态建议定义为：

- `todo` 待开始
- `in_progress` 处理中
- `pending_confirm` 待确认
- `done` 已完成
- `closed` 已关闭
- `overdue` 已逾期

### 状态说明
- 待开始：尚未处理
- 处理中：已开始执行
- 待确认：等待他人确认结果
- 已完成：已完成闭环
- 已关闭：无需继续
- 已逾期：超过截止时间仍未完成

---

## 5.2.5 任务字段

每个任务建议包含以下字段：

- 任务 ID
- 任务类型
- 任务标题
- 任务状态
- 来源订单 ID
- 来源订单商品 ID（可选）
- 责任角色
- 责任人
- 优先级
- 截止时间
- 描述
- 附件占位
- 创建人
- 创建时间
- 更新人
- 更新时间
- 完成人
- 完成时间

---

## 5.2.6 任务创建规则

Phase 2 只支持两种任务来源：

### 来源 1：从订单详情页手动发起
使用场景：
- 跟单发起设计任务
- 客服发起资料补充任务
- 运营发起售后任务

### 来源 2：基于订单状态变化的建议任务
使用场景：
- 订单进入待设计/建模状态时，系统建议创建设计建模任务
- 订单进入售后中状态时，系统建议创建售后任务

### 设计原则
- 本期不做自由复杂任务池
- 任务必须有明确业务来源
- 任务必须能回溯到订单

## 13. 完成标准

满足以下条件，即视为 Phase 2 完成：

1. 可以新建带多个订单商品的订单
2. 订单有明确状态并能完成状态切换
3. 订单详情能展示系统报价和人工调整信息
4. 能从订单创建任务
5. 任务列表页和任务详情页可正常使用
6. 产品详情能展示更完整标准资料
7. 订单详情能展示操作时间线
