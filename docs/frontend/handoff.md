# Handoff（当前阶段说明）

## 1. 文档目标
本文件用于记录当前前端阶段的共识、已确定结论、当前边界和下一轮建议输入，方便：

- 你自己后续继续推进
- Codex / AI 在新对话里快速接上上下文
- 避免前端越做越偏、越做越散

这份文档不是 PRD，也不是任务板。  
它更像当前阶段的**交接说明**。

---

## 2. 当前阶段定位

当前阶段是：

**前端首轮开发准备阶段**

这一阶段的重点不是：

- 后端设计
- 真正接口联调
- 完整业务流闭环
- 复杂权限系统

而是先把这条前端主链路梳理清楚并可直接开工：

**产品维护（含规格明细与固定加价规则）  
→ 订单引用产品  
→ 选择规格  
→ 自动带出规格参数  
→ 自动带出基础价格  
→ 叠加固定加价规则  
→ 生成系统参考报价**

---

## 3. 当前已经明确的核心结论

### 3.1 首轮核心模块
首轮只优先开发两个模块：

1. 订单中心
2. 产品管理

---

### 3.2 产品支持被订单引用
产品不是单纯资料库，而是：

**可被订单引用的标准模板**

订单中的商品可以从产品库中引用产品。

---

### 3.3 订单商品与来源产品是两个对象
必须严格区分：

- 来源产品 = 产品库中的原始模板
- 订单商品 = 订单中的业务实例

订单商品在引用产品后，不与产品模板强绑定。  
订单商品可以在模板基础上继续补充和修改订单实际需求。

---

### 3.4 点击订单中的商品名称
默认打开的是：

**来源产品详情**

不是订单商品详情。

这个能力的目的，是为了在订单执行过程中随时核对：

- 当前订单参数
- 来源产品原始模板参数

是否一致、已调整或冲突。

---

### 3.5 首轮自动报价必须可用
自动带价 / 自动报价已经确认是：

**首轮必做能力**

不能后置到第二轮。

---

### 3.6 首轮只做单轴规格
首轮规格模式只支持：

- 无规格
- 单轴规格

首轮明确不做：

- 多轴规格
- 规格矩阵
- 多维联动价格

---

### 3.7 戒指与吊坠是首轮必须覆盖的样例
首轮必须能演示两条业务样例：

#### 戒指
- 按圈号维护规格
- 订单中选择 16号后自动带参数和价格

#### 吊坠
- 按小 / 中 / 大维护规格
- 订单中选择小号后自动带参数和价格

---

## 4. 当前页面结构共识

## 4.1 订单详情页
订单详情页不是大表单，而是：

1. 顶部订单概览
2. 订单级信息区
3. 商品协同区
4. 物流 / 售后 / 附件 / 日志记录区

---

## 4.2 商品卡
订单商品卡是订单详情页里的核心复用组件。

商品卡当前必须能承载：

- 来源产品条
- 规格与报价区
- 客服需求区
- 设计建模区
- 跟单委外区
- 工厂回传区

首轮真正重点先做：
- 来源产品条
- 规格与报价区

其余区块可以先做结构占位。

---

## 4.3 产品详情页
产品详情页必须采用：

1. 顶部摘要区
2. 分区导航
3. 基础信息
4. 参数配置
5. 价格规则
6. 定制规则
7. 生产参考
8. 图片与文件

---

## 4.4 产品编辑页
产品编辑页必须采用：

1. 顶部保存区
2. 左侧分区导航
3. 右侧分区表单

产品编辑页首轮至少要先做：

- 基础信息
- 参数配置
- 规格明细
- 固定加价规则

---

## 5. 当前业务规则共识

## 5.1 首轮自动报价公式
首轮固定为：

**系统参考报价 = 规格基础价 + 所有生效固定加价之和**

---

## 5.2 固定加价规则类型
首轮只支持：

- 材质
- 工艺
- 特殊需求
- 其他

不做复杂公式和多级优先级。

---

## 5.3 首轮至少支持的 warning
首轮至少提示：

- 未选规格
- 当前规格没有基础价格
- 当前附加项没有对应价格规则

---

## 5.4 首轮暂不做
以下内容已明确后置：

- 人工最终报价
- 多轴规格
- 报价审批流
- 复杂冲突强拦截
- 复杂物流
- 复杂售后
- 高级文件上传管理
- 产品引用记录增强
- 版本历史增强

---

## 5.5 2026-04-22 最新用户反馈（已记录）

### 反馈 1：来源产品详情中的模板原始详情不够完整
用户希望在“查看来源商品详情 -> 模板原始详情”中，不只展示当前选中的规格摘要，而是能一眼看到：

- 每个圈号 / 档位的完整规格参数
- 每个圈号 / 档位的基础价格
- 规格之间的参数差异

建议下一轮增强为：

- 规格参数对照表
- 规格价格矩阵 / 图表化摘要
- 默认高亮当前订单所选规格

当前状态：
- 已能查看来源产品详情
- 但模板原始详情仍偏摘要型，不足以做“所有圈号快速横向核对”

### 反馈 2：新建订单页的基础信息区 / 时间信息区录入方式偏手填
用户希望这些区域尽量不要全部靠人工输入，而是优先支持：

- 下拉选择
- 常用负责人快速选择
- 时间字段更合适的选择方式
- 平台 / 订单类型等字段的默认选项

当前状态：
- 首轮新建订单页仍以基础输入框为主
- 仅完成了页面骨架和主链路演示，录入体验优化还未补齐

### 反馈 3：新建商品页中的“主图链接”语义不清楚
用户对“主图链接”字段的理解成本较高，当前需要明确：

- 该字段现在表示图片 URL / 地址
- 是否支持直接上传图片
- 是否支持从本地选择文件

当前状态：
- 当前实现是手填主图地址
- 还没有做上传图片能力

### 反馈 4：基础信息 / 参数配置中的字段需要默认选项与可维护字典
用户希望以下字段不要只靠自由输入，而是支持：

- 预设默认选项
- 多选 / 标签选择
- 后台可维护选项字典

例如：
- 风格标签：街头风 / 简约 / 时尚 / 国风
- 场景标签
- 支持材质
- 支持工艺
- 特殊需求支持项

当前状态：
- 目前只有少数字段是固定下拉
- `风格标签`、`场景标签`、`支持材质`、`支持工艺` 等仍主要是自由输入
- 后台字典配置能力尚未建立

补充约定（2026-04-22）：
- 当前 mock 阶段允许先做“全局字典模拟”，即：
- 手动新增的选项可以选择加入当前前端全局字典
- 全局字典优先使用 `localStorage` 持久化，保证同一台设备刷新后仍可复用
- 该方案只用于前端演示，不代表真实多人共享能力
- 后续接后端时，必须升级为“多人共享字典”
- 升级后要求：一个人新增并发布的字典选项，其他人在新建 / 编辑商品时也能看到

下拉治理规则补充（2026-04-22）：
- 后续新增的下拉选项，优先按“业务型字典”和“系统型枚举”两类处理
- 业务型字典：
- 例如风格标签、场景标签、材质、工艺、特殊需求、尺寸参数显示名称、单位等
- 这类优先做成管理员或有权限人员可维护的字典
- mock 阶段允许先用前端全局字典 / `localStorage` 模拟
- 接后端后升级为多人共享字典
- 系统型枚举：
- 例如是 / 否、启用 / 禁用、规格模式、规格展示方式等
- 这类保持固定枚举，不建议开放给管理员任意维护
- 后续如果出现新的下拉选择，默认先按这个规则判断，不再一刀切处理

### 反馈 5：界面布局与操作密度需要继续优化
用户提出了 3 条明确的 UI 建议：

- 左侧导航（工作台 / 订单中心 / 产品管理）当前占宽偏大，需要缩窄
- 左侧导航需要固定在可视区域内，页面下拉到底时不能消失
- 新建订单中的“产品引用选择器”底部操作区（确认引用 / 取消）需要固定，避免产品很多时难以找到
- 产品引用选择器中的产品卡片当前尺寸偏大，一屏只能看到很少的产品，影响挑选效率

当前状态：
- 左侧导航目前不是固定侧栏，页面滚动后会影响可见性
- 产品引用选择器底部按钮区当前随内容滚动，不是固定操作区
- 产品选择卡片当前以较大卡片展示为主，信息密度偏低

建议下一轮增强为：
- 左侧导航缩窄并固定，保证页面滚动时始终可见
- `ProductPickerModal` 增加固定底部操作栏
- `ProductPickerCard` 调整为更紧凑的中后台卡片密度，一屏可展示更多产品
