# Handoff（当前阶段说明）

## 1. 文档目标
本文件用于记录当前前端阶段的共识、已确定结论、当前边界和下一轮建议输入，方便：

- 你自己后续继续推进
- Codex / AI 在新对话里快速接上上下文
- 避免前端越做越偏、越做越散

这份文档不是 PRD，也不是任务板。  
它更像当前阶段的**交接说明**。

---

## 2. 当前阶段定位

当前阶段是：

**前端首轮开发准备阶段**

这一阶段的重点不是：

- 后端设计
- 真正接口联调
- 完整业务流闭环
- 复杂权限系统

而是先把这条前端主链路梳理清楚并可直接开工：

**产品维护（含规格明细与固定加价规则）  
→ 商品任务引用产品  
→ 选择规格  
→ 自动带出规格参数  
→ 自动带出基础价格  
→ 叠加固定加价规则  
→ 生成系统参考报价**

---

## 3. 当前已经明确的核心结论

### 3.1 首轮核心模块
首轮只优先开发两个模块：

1. 商品任务中心
2. 产品管理

---

### 3.2 产品支持被商品任务引用
产品不是单纯资料库，而是：

**可被商品任务引用的标准模板**

交易记录中的商品任务可以从产品库中引用产品。

---

### 3.3 商品任务与来源产品是两个对象
必须严格区分：

- 来源产品 = 产品库中的原始模板
- 商品任务 = 交易记录中的业务实例

商品任务在引用产品后，不与产品模板强绑定。  
商品任务可以在模板基础上继续补充和修改本次实际需求。

---

### 3.4 点击商品任务中的商品名称
默认打开的是：

**来源产品详情**

不是商品任务详情。

这个能力的目的，是为了在交易执行过程中随时核对：

- 当前商品任务参数
- 来源产品原始模板参数

是否一致、已调整或冲突。

---

### 3.5 首轮自动报价必须可用
自动带价 / 自动报价已经确认是：

**首轮必做能力**

不能后置到第二轮。

---

### 3.6 首轮只做单轴规格
首轮规格模式只支持：

- 无规格
- 单轴规格

首轮明确不做：

- 多轴规格
- 规格矩阵
- 多维联动价格

---

### 3.7 戒指与吊坠是首轮必须覆盖的样例
首轮必须能演示两条业务样例：

#### 戒指
- 按圈号维护规格
- 商品任务中选择 16号后自动带参数和价格

#### 吊坠
- 按小 / 中 / 大维护规格
- 商品任务中选择小号后自动带参数和价格

## 5. 当前业务规则共识

首轮固定为：

**系统参考报价 = 规格基础价 + 所有生效固定加价之和**

## 13. 2026-04-23 客服角色字段口径补充

用户已明确：`基础信息 / 客户信息 / 时间信息` 只是梳理分类，后续实现必须以具体字段语义为准，不能因为页面上已经有同名分区卡片，就误判为客服角色已经覆盖完成。

### 13.1 已确认字段语义

#### 接待客服
- 业务语义等同于当前前端里的 `客服负责人`
- 后续优先调整页面展示文案为 `接待客服`

#### 订单登记时间
- 新建订单时系统自动生成
- 不需要客服手动输入
- 订单详情中需要明确展示，并作为可追溯记录保留

#### 平台ID
- 必须作为独立字段存在
- 不能只保留 `客户姓名`
- 原因：部分平台聊天工具主要通过 `平台ID` 识别客户，而收货信息里的姓名不一定能帮助客服快速定位聊天对象

#### 订单类型
- 订单类型后续统一直接区分为：
  - `销售订单`
  - `内部订单`
- 其中：
  - `销售订单` 覆盖平台销售、私域销售、线下销售等所有对外销售场景
  - `内部订单` 覆盖企业研发、内部设计等内部流转场景

#### 平台订单编号
- 保持现有命名，不改成“外部订单编号”
- 该字段继续存在，主要用于承接平台侧或对外销售侧单号
- 与 `订单类型`、`来源渠道` 配合理解

#### 来源渠道
- 例如：淘宝、京东、小红书、抖音、微信等
- 需要做成可选字典
- 字典延续当前项目规则：
  - mock 阶段先做前端可维护字典
  - 接后端后升级为多人共享字典

#### 订单类型与角色权限关系
- `销售订单`
  - 可由客服、管理层、运营等角色填写
- `内部订单`
  - 仅允许设计、管理员填写
- 用户已确认：`销售` 不单独作为独立角色，直接并入 `客服`
- 这意味着后续订单编辑权限不能只按“角色”判断，还要叠加“订单类型”判断
- 当前前端 mock 角色里后续仍需要补独立 `管理员`

#### 时间字段必须区分 3 个概念
- `客户期望时间`
- `计划交期`
- `承诺交期`

其业务语义已确认：
- 客户期望时间：客户希望拿到货的时间
- 计划交期：内部根据款式 / 工艺 / 节假日 / 缓冲预留推算出的计划完成或计划寄出时间
- 承诺交期：对客户明确承诺的寄出或交付时间

示例：
- 客户 1 号下单定制戒指，期望 15 号拿到
- 该款工期预计 7 天
- 则内部 `计划交期` 可为 8 号
- 对客户 `承诺交期` 可为 10 号

后续系统能力要求：
- 支持根据工期逻辑自动给出 `计划交期`
- 清楚展示 `承诺交期`
- 校验 `计划交期 / 承诺交期 / 客户期望时间` 是否冲突

#### 财务信息
- 当前商品卡里的 `系统参考报价` 只是系统自动带出的参考价
- 它不是客服最终填写或确认的成交价格
- 财务信息后续至少要补：
  - `定金`
  - `尾款`
  - `退差 / 补偿`
- 业务原因：
  - 客户第一次付款时可能只支付定金
  - 后续需要根据实际成交情况做尾款结算
  - 如果实际价格变化，还需要多退少补

#### 财务信息不能只靠 3 个静态字段
用户已进一步确认：一个订单后续可能出现多次付款、平台退款、线下退款、售后补款、售后退款等动作。

示例链路：
- 系统参考价：10000
- 活动成交价：8500
- 定金：5000
- 尾款：3500
- 发货后平台退差：100
- 确认收货后好评返现退款：100
- 后续非平台退款：100
- 售后补款：500
- 售后退款：200

由此可得：
- `系统参考价` 与 `实际成交价` 必须分开
- 仅有 `定金 / 尾款 / 退差补偿` 不足以覆盖完整财务过程

后续前端更合理的数据口径应为两层：

##### 第一层：交易记录财务摘要字段
- `systemReferenceQuote` 系统参考价
- `actualDealAmount` 实际成交价 / 活动成交价
- `depositPaid` 定金
- `balancePaid` 尾款
- `receivedAmount` 累计收款
- `refundedAmount` 累计退款
- `netPaidAmount` 累计净收款
- `invoiceIssued` 是否开过发票
- `financeRemark` 财务备注

##### 第二层：交易记录财务流水
用于逐条记录动作，至少支持：
- `deposit_payment` 定金收款
- `balance_payment` 尾款收款
- `platform_refund` 平台退款
- `manual_refund` 线下退款 / 非平台退款
- `after_sales_payment` 售后补款
- `after_sales_refund` 售后退款

计算口径建议：
- `receivedAmount` = 所有收款流水之和
- `refundedAmount` = 所有退款流水之和
- `netPaidAmount` = `receivedAmount - refundedAmount`

注意：
- 如果业务上把“实际付款金额”定义为“活动成交后客户最终承担金额”，则该值更接近 `actualDealAmount`
- 如果定义为“当前累计净收款”，则更接近 `netPaidAmount`
- 后续页面文案必须选定一种口径，避免混淆
- 当前用户已确认，这里优先采用：
  - `实际成交价`
  - `累计净收款`

### 13.2 对后续前端实现的直接影响

客服角色后续补字段时，优先顺序调整为：
1. 补 `平台ID / 来源渠道 / 订单登记时间 / 计划交期`
2. 把 `客服负责人` 的展示文案统一到 `接待客服`
3. 把 `订单类型` 收口为 `销售订单 / 内部订单`
4. 保留并补强 `平台订单编号`
5. 再扩展财务摘要字段与财务流水
6. 最后补商品需求采集与售后信息

## 14. 生产计划薄页面说明

### 14.1 当前落位
当前仓库允许新增一个实验性的独立薄页面：

- `/production-plan`
- `/production-plan/:taskId`

它的定位不是新的一级业务中心，也不是完整工厂协同模块，而是：

**基于现有 `factory_production` mock 任务的工厂演示视图。**

---

### 14.2 当前边界

这个页面当前只做：

- 列表查看
- 详情查看
- 轻量状态操作
- 工厂反馈 / 回传摘要演示
- 商品任务级追溯

这个页面当前不做：

- 独立权限系统
- 独立工厂状态机
- 真实附件上传
- 真实后端回传提交流程
- 物流 / 财务 / 成本扩展
- 完整工厂协同中心

---

### 14.3 数据口径

页面只消费这些现有对象：

- `Task`
- `Order`
- `OrderItem`
- `Product`

并通过前端适配层生成页面专用 view model。

注意：

- `factory_production` 仍然只作为当前页面的隔离 mock 任务类型使用
- 它不代表本轮要把全项目正式升级为工厂协同模式
- 订单状态体系、任务状态体系和主业务链路不因这个页面而改名或重做

---

### 14.4 当前实现原则

- 页面只在 `factory` 角色的侧边栏中暴露入口
- 非 `factory` 角色仍可通过直达路由查看页面，但不做路由守卫
- 所有操作继续走现有 `updateTask` / `updateOrderItem`
- 交易记录时间线仍然是当前唯一追溯主线之一，不额外新建第二套流程引擎

---

### 14.5 对后续迭代的建议

如果后续要继续往“真正工厂页”推进，建议顺序为：

1. 先补稳定的工厂展示字段与前端派生对象
2. 再补工厂列表 / 详情的更多回传分组
3. 再决定是否进入独立状态机和独立权限
4. 最后才考虑完整工厂协同中心

不要直接从当前薄页面跳到完整工厂系统。
