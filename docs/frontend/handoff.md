## 3. 历史首轮核心结论（保留参考，非当前命名口径）

### 3.1 历史首轮核心模块
首轮只优先开发两个模块：

1. 销售中心
2. 款式管理

---

### 3.2 产品支持被销售引用
产品不是单纯资料库，而是：

**可被销售引用的标准模板**

购买记录中的销售可以从款式库中引用款式。

---

### 3.3 销售与来源款式是两个对象
必须严格区分：

- 来源款式 = 款式库中的原始模板
- 销售 = 购买记录中的单件商品执行对象

销售在引用款式后，不与款式模板强绑定。
销售可以在模板基础上继续补充和修改本次实际需求。

---

### 3.4 点击销售中的款式名称
默认打开的是：

**来源款式详情**

不是购买记录详情。

这个能力的目的，是为了在交易执行过程中随时核对：

- 当前销售参数
- 来源款式原始模板参数

是否一致、已调整或冲突。

---

### 3.5 首轮自动报价必须可用
自动带价 / 自动报价已经确认是：

**首轮必做能力**

不能后置到第二轮。

---

### 3.6 首轮只做单轴规格
首轮规格模式只支持：

- 无规格
- 单轴规格

首轮明确不做：

- 多轴规格
- 规格矩阵
- 多维联动价格

---

### 3.7 戒指与吊坠是首轮必须覆盖的样例
首轮必须能演示两条业务样例：

#### 戒指
- 按圈号维护规格
- 商品任务中选择 16号后自动带参数和价格

#### 吊坠
- 按小 / 中 / 大维护规格
- 商品任务中选择小号后自动带参数和价格

## 4. 历史页面结构共识（保留参考，当前以 routes / ui 文档为准）

2. 交易级信息区
3. 商品协同区

## 4.2 商品卡
商品任务卡是交易记录详情页里的核心复用组件。

商品卡当前必须能承载：

- 来源款式条
- 规格与报价区
- 客服需求区
- 设计建模区
- 跟单委外区
- 工厂回传区

首轮真正重点先做：
- 来源款式条
- 规格与报价区

其余区块可以先做结构占位。

## 4.3 款式详情页
款式详情页必须采用：

1. 顶部摘要区
2. 分区导航
3. 基础信息
4. 参数配置
5. 价格规则
6. 定制规则
7. 生产参考
8. 图片与文件

## 5. 当前业务规则共识

## 5.1 首轮自动报价公式
首轮固定为：

**系统参考报价 = 规格基础价 + 所有生效固定加价之和**

---

## 5.2 固定加价规则类型
首轮只支持：

- 材质
- 工艺
- 特殊需求
- 其他

不做复杂公式和多级优先级。

---

## 5.3 首轮至少支持的 warning
首轮至少提示：

- 未选规格
- 当前规格没有基础价格
- 当前附加项没有对应价格规则

## 5.5 2026-04-22 最新用户反馈（已记录）

### 反馈 1：来源款式详情中的模板原始详情不够完整
用户希望在“查看来源商品详情 -> 模板原始详情”中，不只展示当前选中的规格摘要，而是能一眼看到：

- 每个圈号 / 档位的完整规格参数
- 每个圈号 / 档位的基础价格
- 规格之间的参数差异

建议下一轮增强为：

- 规格参数对照表
- 规格价格矩阵 / 图表化摘要
- 默认高亮当前订单所选规格

当前状态：
- 已能查看来源款式详情
- 但模板原始详情仍偏摘要型，不足以做“所有圈号快速横向核对”

### 反馈 2：新建交易记录页的基础信息区 / 时间信息区录入方式偏手填
用户希望这些区域尽量不要全部靠人工输入，而是优先支持：

- 下拉选择
- 常用负责人快速选择
- 时间字段更合适的选择方式
- 平台 / 订单类型等字段的默认选项

当前状态：
- 首轮新建交易记录页仍以基础输入框为主
- 仅完成了页面骨架和主链路演示，录入体验优化还未补齐

### 反馈 3：新建商品页中的“主图链接”语义不清楚
用户对“主图链接”字段的理解成本较高，当前需要明确：

- 该字段现在表示图片 URL / 地址
- 是否支持直接上传图片
- 是否支持从本地选择文件

当前状态：
- 当前实现是手填主图地址
- 还没有做上传图片能力

### 反馈 4：基础信息 / 参数配置中的字段需要默认选项与可维护字典
用户希望以下字段不要只靠自由输入，而是支持：

- 预设默认选项
- 多选 / 标签选择
- 后台可维护选项字典

例如：
- 风格标签：街头风 / 简约 / 时尚 / 国风
- 场景标签
- 支持材质
- 支持工艺
- 特殊需求支持项

当前状态：
- 目前只有少数字段是固定下拉
- `风格标签`、`场景标签`、`支持材质`、`支持工艺` 等仍主要是自由输入
- 后台字典配置能力尚未建立

补充约定（2026-04-22）：
- 当前 mock 阶段允许先做“全局字典模拟”，即：
- 手动新增的选项可以选择加入当前前端全局字典
- 全局字典优先使用 `localStorage` 持久化，保证同一台设备刷新后仍可复用
- 该方案只用于前端演示，不代表真实多人共享能力
- 后续接后端时，必须升级为“多人共享字典”
- 升级后要求：一个人新增并发布的字典选项，其他人在新建 / 编辑商品时也能看到

下拉治理规则补充（2026-04-22）：
- 后续新增的下拉选项，优先按“业务型字典”和“系统型枚举”两类处理
- 业务型字典：
- 例如风格标签、场景标签、材质、工艺、特殊需求、尺寸参数显示名称、单位等
- 这类优先做成管理员或有权限人员可维护的字典
- mock 阶段允许先用前端全局字典 / `localStorage` 模拟
- 接后端后升级为多人共享字典
- 系统型枚举：
- 例如是 / 否、启用 / 禁用、规格模式、规格展示方式等
- 这类保持固定枚举，不建议开放给管理员任意维护
- 后续如果出现新的下拉选择，默认先按这个规则判断，不再一刀切处理

### 反馈 5：界面布局与操作密度需要继续优化
用户提出了 3 条明确的 UI 建议：

- 左侧导航（工作台 / 销售中心 / 款式管理）当前占宽偏大，需要缩窄
- 左侧导航需要固定在可视区域内，页面下拉到底时不能消失
- 新建交易记录页中的“产品引用选择器”底部操作区（确认引用 / 取消）需要固定，避免产品很多时难以找到
- 产品引用选择器中的产品卡片当前尺寸偏大，一屏只能看到很少的产品，影响挑选效率

当前状态：
- 左侧导航目前不是固定侧栏，页面滚动后会影响可见性
- 产品引用选择器底部按钮区当前随内容滚动，不是固定操作区
- 产品选择卡片当前以较大卡片展示为主，信息密度偏低

建议下一轮增强为：
- 左侧导航缩窄并固定，保证页面滚动时始终可见
- `ProductPickerModal` 增加固定底部操作栏
- `ProductPickerCard` 调整为更紧凑的中后台卡片密度，一屏可展示更多产品

## 13. 2026-04-23 客服角色字段口径补充

用户已明确：`基础信息 / 客户信息 / 时间信息` 只是梳理分类，后续实现必须以具体字段语义为准，不能因为页面上已经有同名分区卡片，就误判为客服角色已经覆盖完成。

### 13.1 已确认字段语义

#### 接待客服
- 业务语义等同于当前前端里的 `客服负责人`
- 后续优先调整页面展示文案为 `接待客服`

#### 购买记录登记时间
- 新建购买记录时系统自动生成
- 不需要客服手动输入
- 购买记录详情中需要明确展示，并作为可追溯记录保留

#### 平台ID
- 必须作为独立字段存在
- 不能只保留 `客户姓名`
- 原因：部分平台聊天工具主要通过 `平台ID` 识别客户，而收货信息里的姓名不一定能帮助客服快速定位聊天对象

#### 订单类型
- 订单类型后续统一直接区分为：
  - `销售订单`
  - `内部订单`
- 其中：
  - `销售订单` 覆盖平台销售、私域销售、线下销售等所有对外销售场景
  - `内部订单` 覆盖企业研发、内部设计等内部流转场景

#### 平台订单编号
- 保持现有命名，不改成“外部订单编号”
- 该字段继续存在，主要用于承接平台侧或对外销售侧单号
- 与 `订单类型`、`来源渠道` 配合理解

#### 来源渠道
- 例如：淘宝、京东、小红书、抖音、微信等
- 需要做成可选字典
- 字典延续当前项目规则：
  - mock 阶段先做前端可维护字典
  - 接后端后升级为多人共享字典

#### 订单类型与角色权限关系
- `销售订单`
  - 可由客服、管理层、运营等角色填写
- `内部订单`
  - 仅允许设计、管理员填写
- 用户已确认：`销售` 不单独作为独立角色，直接并入 `客服`
- 这意味着后续订单编辑权限不能只按“角色”判断，还要叠加“订单类型”判断
- 当前前端 mock 角色里后续仍需要补独立 `管理员`

#### 时间字段必须区分 3 个概念
- `客户期望时间`
- `计划交期`
- `承诺交期`

其业务语义已确认：
- 客户期望时间：客户希望拿到货的时间
- 计划交期：内部根据款式 / 工艺 / 节假日 / 缓冲预留推算出的计划完成或计划寄出时间
- 承诺交期：对客户明确承诺的寄出或交付时间

示例：
- 客户 1 号下单定制戒指，期望 15 号拿到
- 该款工期预计 7 天
- 则内部 `计划交期` 可为 8 号
- 对客户 `承诺交期` 可为 10 号

后续系统能力要求：
- 支持根据工期逻辑自动给出 `计划交期`
- 清楚展示 `承诺交期`
- 校验 `计划交期 / 承诺交期 / 客户期望时间` 是否冲突

#### 财务信息
- 当前商品卡里的 `系统参考报价` 只是系统自动带出的参考价
- 它不是客服最终填写或确认的成交价格
- 财务信息后续至少要补：
  - `定金`
  - `尾款`
  - `退差 / 补偿`
- 业务原因：
  - 客户第一次付款时可能只支付定金
  - 后续需要根据实际成交情况做尾款结算
  - 如果实际价格变化，还需要多退少补

#### 财务信息不能只靠 3 个静态字段
用户已进一步确认：一个订单后续可能出现多次付款、平台退款、线下退款、售后补款、售后退款等动作。

示例链路：
- 系统参考价：10000
- 活动成交价：8500
- 定金：5000
- 尾款：3500
- 发货后平台退差：100
- 确认收货后好评返现退款：100
- 后续非平台退款：100
- 售后补款：500
- 售后退款：200

由此可得：
- `系统参考价` 与 `实际成交价` 必须分开
- 仅有 `定金 / 尾款 / 退差补偿` 不足以覆盖完整财务过程

后续前端更合理的数据口径应为两层：

##### 第一层：交易记录财务摘要字段
- `systemReferenceQuote` 系统参考价
- `actualDealAmount` 实际成交价 / 活动成交价
- `depositPaid` 定金
- `balancePaid` 尾款
- `receivedAmount` 累计收款
- `refundedAmount` 累计退款
- `netPaidAmount` 累计净收款
- `invoiceIssued` 是否开过发票
- `financeRemark` 财务备注

##### 第二层：交易记录财务流水
用于逐条记录动作，至少支持：
- `deposit_payment` 定金收款
- `balance_payment` 尾款收款
- `platform_refund` 平台退款
- `manual_refund` 线下退款 / 非平台退款
- `after_sales_payment` 售后补款
- `after_sales_refund` 售后退款

计算口径建议：
- `receivedAmount` = 所有收款流水之和
- `refundedAmount` = 所有退款流水之和
- `netPaidAmount` = `receivedAmount - refundedAmount`

注意：
- 如果业务上把“实际付款金额”定义为“活动成交后客户最终承担金额”，则该值更接近 `actualDealAmount`
- 如果定义为“当前累计净收款”，则更接近 `netPaidAmount`
- 后续页面文案必须选定一种口径，避免混淆
- 当前用户已确认，这里优先采用：
  - `实际成交价`
  - `累计净收款`

### 13.2 对后续前端实现的直接影响
