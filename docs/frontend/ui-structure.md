# 页面结构总览

## 目标
本文件只约束当前 `Purchase + OrderLine` 主线的 UI 信息层级。旧阶段 UI 草案已归档。

## 总体原则
- `OrderLine` 是主操作对象，一件商品一行。
- `Purchase` 是归组对象，展示一次购买的公共信息和多条商品行。
- `Product` 是模板对象，不进入执行流转。
- 物流和售后默认按 `orderLineId` 展示。
- legacy `/orders` 不再作为 UI 入口或兼容入口。

## `/order-lines` 商品行中心
页面目标：快速查看和推进单件商品执行状态。

结构：
1. 页面标题区：标题、关键统计、新建购买记录入口。
2. 筛选区：关键词、`lineStatus`、负责人、交期、生产状态、工厂状态。
3. 商品行表格：商品行编号、购买记录、客户摘要、产品名、规格、主状态、角色状态、负责人、交期。
4. 商品行详情抽屉：展示来源产品、实际需求、报价、设计 / 建模 / 生产 / 工厂 / 财务摘要。

边界：
- 不把一行展示成整笔购买。
- 不读取旧 `OrderItem`。

## `/purchases/new` 新建购买记录
页面目标：创建一次购买及其多条商品行草稿。

结构：
1. 购买公共信息：平台、平台订单号、购买时间、备注。
2. 客户与收件信息：客户选择、联系人、地址。
3. 付款汇总：定金、尾款、优惠、应收。
4. 商品行草稿区：每张卡代表一条未来 `OrderLine`。
5. 提交区：保存 `purchaseDraft + orderLineDrafts`。

商品行草稿卡至少包含：
- 来源产品选择
- 规格选择
- 实际材质、尺寸、工艺、印记、特殊需求
- 系统参考报价
- 资料完整度
- `requiresDesign / requiresModeling`

## `/purchases/:purchaseId` 购买记录详情
页面目标：查看一次购买的公共信息和本次购买下的所有商品行。

结构：
1. 购买记录摘要：平台订单号、客户、付款、聚合状态。
2. 客户与收件信息。
3. 付款总览。
4. 本次商品行列表：多条 `OrderLine` 独立展示状态、交期、负责人、物流和售后入口。
5. 购买记录时间线：只展示归组层面的事件。

边界：
- 不承接单件商品的完整执行页职责。
- 不用 `Purchase.aggregateStatus` 驱动单件状态流转。

## `/products` 产品管理
页面目标：维护可被商品行引用的产品模板。

列表页展示：
- 产品名、分类、SKU / 货号、款式、版本、状态、基础价格摘要。

详情页展示：
- 基础信息
- 参数配置
- 规格明细
- 固定加价规则
- 定制规则
- 生产参考
- 文件资料

编辑页用于维护模板字段，不直接修改已存在商品行的执行数据。

## 来源产品详情抽屉
触发：商品行里的产品名称或来源产品条。

结构：
1. 产品模板摘要。
2. 模板规格与价格规则。
3. 商品行实际选择。
4. 模板参数和实际参数对比。

用途：帮助核对商品行是否偏离来源模板。

## 角色工作台信息边界
- 生产跟进：显示跟单推进必需字段和生产风险。
- 设计 / 建模：显示执行需求、设计 / 建模状态和文件，不显示客户隐私或财务金额。
- 工厂协同：显示生产资料和回传字段，不显示客户隐私、地址、销售金额、利润或财务备注。
- 财务中心：显示金额、成本和工厂回传数据，不编辑设计、建模或生产字段。
- 管理看板：只展示聚合指标和 drill-down 入口。

## 验收标准
- 商品行中心、购买记录详情、角色工作台都以 `OrderLine` 为执行单位。
- 新建购买记录能创建多条商品行草稿。
- 产品页只展示模板对象。
- 同一次购买中的商品行能分别发货、分别售后。
- 页面不再出现 legacy `/orders` 入口。
